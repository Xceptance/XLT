# Cached Requests Logging in XLT

This document describes the implementation details and user-facing changes for the Cached Requests Logging feature introduced in XLT 10.0.0.

## 1. End-User Documentation (for docs.xceptance.com/xlt)

XLT now tracks and logs browser-cached requests during a load test. Previously, when the browser (HtmlUnit) served a resource directly from its internal cache without making a network request (e.g., due to `Cache-Control` headers), these requests were completely invisible in the XLT reports and result browser.

With this feature enabled, cache hits are explicitly logged and visualized:
- **Result Browser:** Cached requests appear in the request list, greyed out and marked with "(from cache)".
- **Reports:** You can now filter or group cached requests using Report Merge Rules.
- **HAR Export:** HAR files generated by XLT will include these requests, accurately reflecting their cached status.

### 1.1 Configuration

Logging of cached requests is enabled by default. You can disable it by setting the following property in your `project.properties` or `default.properties`:

```properties
com.xceptance.xlt.http.cachedRequests.logging = false
```

### 1.2 Report Merge Rules

A new placeholder `{cache}` is available for report merge rules. This allows you to specifically include, exclude, or group requests based on whether they were served from the cache.

To apply rules specifically to cached requests, use the `cachedPattern` property:

```properties
# Exclude all cached requests from being merged into this rule
com.xceptance.xlt.reportgenerator.rules.myRule.cachedPattern.exclude = true

# Only match requests that were served from cache
com.xceptance.xlt.reportgenerator.rules.myRule.cachedPattern = true
```

#### Backward Compatibility with Older Test Reports
When processing test reports generated by older XLT versions (pre-10.0.0), the report generator safely assumes that all requests have `cached = false`. This guarantees backward compatibility:
- A new merge rule looking for `cachedPattern = true` will simply match zero requests in an old report.
- The `{cache}` placeholder will correctly resolve to `"false"` for all requests in older data sets.
Because older versions of XLT only logged network-level cache misses anyway, treating all old data as "not cached" is logically correct and will not break any existing rules or workflows.

### 1.3 Metrics Impact

When a request is served from the cache, no actual network traffic occurs. Therefore, the logged metrics for this request will have `0` for all time-based metrics (connect time, send time, receive time, DNS time, etc.) and `0` for `bytesSent`. The `bytesReceived` metric will reflect the uncompressed size of the cached payload, ensuring page size statistics remain accurate.

## 2. Developer Documentation (Architecture & Implementation)

This section provides technical details for XLT developers maintaining or extending this feature.

### 2.1 Core Interception Hook

The core interception point is in the HTTP connection layer.
- `CachingHttpWebConnection` (the base class) has been extended with a protected template method hook: `protected void reportCacheHit(WebRequest, WebResponse)`.
- `XltHttpWebConnection` overrides this method. When a cache hit occurs, it creates a new `RequestData` object, populates it with zeroed metrics (since no network transfer happened), sets the new `cached` flag to `true`, and logs the record via `Session.getCurrent().getDataManager().logDataRecord()`.
- **Note:** `NetworkData` is intentionally NOT logged for cache hits, as no actual network traffic took place.

### 2.2 Data Model (`RequestData`)

A new boolean field `cached` was added to `RequestData`. 
To maintain backward compatibility with older XLT versions and report generators, the CSV serialization/deserialization logic handles this new field cleanly:
- It is appended to the end of the CSV line.
- The CSV parser (`remainingValuesFromCSV` / `setRemainingValues`) checks the number of columns (`if (values.size() > 24)`) to determine if the `cached` flag is present, defaulting to `false` for older data files.

### 2.3 Report Generator and Merge Rules

The merge rule system was enhanced to support the `{cache}` placeholder:
- `MergeRule`'s `PLACEHOLDER_PATTERN` regex was updated to `([acmnrstu]|cache)`.
- A new condition type was added: `CachedCondition` (and `CachedEmptyCondition`). It behaves similarly to `HttpMethodCondition` or `StatusCodeCondition`, filtering requests based on their `isCached()` state.
- `RequestDataProcessor` handles the `RequestData` objects normally, correctly aggregating the zeroed metrics without skewing latency averages.

### 2.4 Result Browser (`DumpMgr`)

The `DumpMgr` is responsible for exporting request payloads and generating the JSON data for the Result Browser.
- **Payload Dumping:** `DumpMgr` explicitly checks `requestData.isCached()`. If a request is cached, `DumpMgr` skips writing the payload content to the `responses` directory, saving disk space and I/O.
- **Data Model:** The `RequestInfo` object used to generate `data.js` now includes the `cached` boolean.
- **HAR Export:** `HarExporter` maps the `cached` flag to the HAR-standard `cache` object (specifically `beforeRequest` / `afterRequest` logic, though simplified for XLT's context) so external tools recognize them as cache hits.

### 2.5 Result Browser UI (JavaScript/CSS)

The Result Browser frontend was updated to visualize cache hits:
- `resultbrowser.js` reads the `cached` property.
- For cached requests, it appends the `contentTypeCached` CSS class to the request entry in the left navigation tree, turning it grey and italic.
- The response headers tab displays a "Response body not stored for cached requests" message instead of a generic "No content" message or attempting to fetch a non-existent payload file.
- The status line indicates " (from cache)".
